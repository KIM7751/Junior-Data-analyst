        [4] 객체지향 방법론 - 현실의 실체에 대한 데이터와 기능을 컴퓨터 즉, 객체로 구현하고자 함
                                   
                            - 방법론의 절차로는 폭포수 모형과 다르게 "요구분석 -> 설계 -> 구현" 단계가 명확하게 구분 되지 않음
                            
                            - 데이터와 데이터에 관련되는 동작을 모두 포함
                            
                            - 데이터는 실체, 동작은 절차, 방법, 기능을 의미
                            
                            - 정보 시스템과 데이터베이스를 설계하는 방법론
                            
                            - 객체 중심으로 캡슐화(객체화), 추상화(사람, 새, 말 -> 동물이라는 대표성을 가진 추상적인 데이터) 기술이 필요
                            
                            - 분석 초점이 명확
                            
                            - 자연스럽고 유연하며 "재사용"이 용이하다
                            
                            - 개발 전문가 부족
                            
                            
        [5] 컴포넌트기반 방법론 - 소프트웨어를 구성하는 컴포넌트를 조립해 하나의 새로운 애플리케이션을 작성함
                                
                                - 모듈은 기능을 구현하기 위한 최소의 단위
                                
                                - 공공 행정 정보 시스템의 개발에 많이 활용되고 있는 표준 프로세스
                                
                                - 재사용이 가능한 컴포넌트 개발 또는 상용 컴포넌트들을 조합해 개발
                                
                                - 생산성과 품질을 높이고, 유지보수의 비용(오류, 수정)을 최소화할 수 있는 개발
                                
                                - 반복적, 점진적 개발
                                
                                - 재사용성 생산성, 품질이 높은 방법
                                
                                - 비용 저렴, 위험 개선
                                
                                - 소프트웨어 위기를 극복하기 위한 방법
                                
                                - 컴포넌트 유통의 환경을 개선해야 함(내가 쓸 라이브러리는 5000개 라이브러리가 포함된 컴포넌트 중 50개 -> 비효율)
                                
                                - 테스트 환경이 부족, 컴포넌트 평가, 인증 환경이 미흡(관공서 외에는 다양하게 써보지 않음, S/W 특성상)
